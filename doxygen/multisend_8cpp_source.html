7 /*
8 Overall exchange strategy
9 
10 When a cell spikes, it immediately does a multisend of
11 (int gid, double spiketime) to all the target machines that have
12 cells that need to receive this spike by spiketime + delay.
13 The MPI implementation does not block due to use of MPI_Isend.
14 
15 In order to minimize the number of nrnmpi_multisend_conserve tests
16 (and potentially abandon them altogether if I can ever guarantee
17 that exchange time is less than half the computation time), I divide the
18 minimum delay integration intervals into two equal subintervals.
19 So if a spike is generated in an even subinterval, I do not have
20 to include it in the conservation check until the end of the next even
21 subinterval.
22 
23 When a spike is received (generally MPI_Iprobe, MPI_Recv) it is placed in
24 even or odd buffers (depending on whether the coded gid is positive or negative)
25 
26 At the end of a computation subinterval the even or odd buffer spikes
27 are enqueued in the priority queue after checking that the number
28 of spikes sent is equal to the number of spikes sent.
29 */

31 // The initial idea behind the optional phase2 is to avoid the large overhead of
32 // initiating a send of the up to 10k list of target hosts when a cell fires.
33 // I.e. when there are a small number of cells on a processor, this causes
34 // load balance problems.
35 // Load balance should be better if the send is distributed to a much smaller
36 // set of targets, which, when they receive the spike, pass it on to a neighbor
37 // set. A non-exclusive alternative to this is the use of RECORD_REPLAY
38 // which give a very fast initiation but we have not been able to get that
39 // to complete in the sense of all the targets receiving their spikes before
40 // the conservation step.
41 // We expect that phase2 will work best in combination with ENQUEUE=2
42 // which has the greatest amount of overlap between computation
43 // and communication.
